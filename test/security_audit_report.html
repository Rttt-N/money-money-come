🔴 High Severity
H-1 HIGH USDC Blacklist Causes Settlement to Permanently Freeze ▼
MoneyMoneyCome.sol : 363–366 Category: DoS / Fund Lockup Function: fulfillRandomWords()

Description

USDC supports address blacklisting (OFAC compliance). When a squad member wins, the contract loops through all squad members and immediately transfers USDC to each one. If any member address is blacklisted, safeTransfer reverts — and since this happens inside the Chainlink VRF callback, the entire callback transaction reverts.

// MoneyMoneyCome.sol lines 363–366
for (uint256 i; i < otherMembers.length; i++) {
    if (otherAmounts[i] > 0)
        usdc.safeTransfer(otherMembers[i], otherAmounts[i]);  // ← any failure reverts ALL
}

Impact

The round is permanently stuck in DRAWING state. All participants' principal and the entire prize pool are locked in the contract with no recovery path. An attacker can deliberately engineer this condition.

Recommendation — Pull Payment Pattern

// Replace direct transfers with a claimable balance
mapping(address => uint256) public pendingWithdrawals;

// In fulfillRandomWords(), replace safeTransfer calls with:
pendingWithdrawals[winner] += winnerAmount;
for (uint256 i; ...) {
    pendingWithdrawals[otherMembers[i]] += otherAmounts[i];
}

// New function — each recipient claims their own prize
function claimPrize() external nonReentrant {
    uint256 amount = pendingWithdrawals[msg.sender];
    require(amount > 0, "Nothing to claim");
    pendingWithdrawals[msg.sender] = 0;
    usdc.safeTransfer(msg.sender, amount);
}
H-2 HIGH Division by Zero in Winner Selection if All Users Withdraw During DRAWING ▼
MoneyMoneyCome.sol : 331 Category: Arithmetic / DoS Function: fulfillRandomWords()

Description

Between performUpkeep() and the VRF callback, the round is in DRAWING state. Users can still call withdraw() during this window, which decrements rounds[currentRound].totalWeight. If all participants withdraw, totalWeight reaches 0. When the VRF callback fires:

// MoneyMoneyCome.sol line 331
uint256 rand = randomWords[0] % r.totalWeight;
//                            ↑ Solidity 0.8 panic if totalWeight == 0

Impact

The VRF callback reverts (Solidity 0.8 panic code 0x12 on modulo by zero). The round is permanently stuck in DRAWING state. The prize pool yield is locked forever.

Recommendation

// Add before the modulo operation in fulfillRandomWords()
if (r.totalWeight == 0) {
    r.state = RoundState.SETTLED;
    rounds[currentRound + 1].prizePool += prize; // carry forward
    _startNewRound();
    return;
}
H-3 HIGH performUpkeep() Can Be Called on Empty Round, Freezing the Protocol ▼
MoneyMoneyCome.sol : 295–317 Category: Missing Guard / DoS Function: performUpkeep()

Description

checkUpkeep() correctly requires participants.length > 0, but performUpkeep() has no such check. Anyone can call it directly once a round has expired, even with zero participants.

// checkUpkeep() — has guard ✓
upkeepNeeded = (block.timestamp >= r.endTime
    && r.state == RoundState.OPEN
    && _roundParticipants[currentRound].length > 0);  ← guard exists here

// performUpkeep() — NO guard ✗
function performUpkeep(bytes calldata) external override {
    require(block.timestamp >= r.endTime, ...);
    require(r.state == RoundState.OPEN, ...);
    // ← participants.length check is MISSING
    r.state = RoundState.LOCKED;
    ...
    vrfCoord.requestRandomWords(req);          // VRF requested
    rounds[currentRound].state = RoundState.DRAWING;
}

When the VRF callback arrives for an empty round:

require(participants.length > 0, "MMC: no participants");  ← REVERTS
// Round permanently stuck in DRAWING state

Recommendation

function performUpkeep(bytes calldata) external override {
    require(block.timestamp >= r.endTime, "MMC: round not ended");
    require(r.state == RoundState.OPEN,   "MMC: wrong state");
    require(
        _roundParticipants[currentRound].length > 0,
        "MMC: no participants"                          ← ADD THIS
    );
    ...
}
🟠 Medium Severity
M-1 MEDIUM NFT Transfer Followed by Top-Up Permanently Locks the User Out ▼
MoneyMoneyCome.sol : 204–210 TicketNFT.sol : 99–107 Category: Logic Error / DoS

Description

If a user transfers their TicketNFT to another address (Alice) and then calls enterGame() again to top up, the burn logic uses the ERC-721 ownerOf() to find the owner (which is now Alice), and clears Alice's userRoundTicket mapping. The original user's mapping is never cleared, so the subsequent mint() fails.

// User transfers NFT to Alice, then calls enterGame() again

// Step 1: lookup finds old tokenId (mapping not updated by ERC-721 transfer)
existingTicket = ticketNFT.userRoundTicket(msg.sender, currentRound);  // = 5

// Step 2: burn(5) — owner is now Alice
address owner = ownerOf(5);                           // = Alice
delete userRoundTicket[Alice][roundId];             // ← Alice's mapping cleared
// userRoundTicket[msg.sender][roundId] still = 5 !

// Step 3: mint check
require(userRoundTicket[msg.sender][roundId] == 0);  // ← REVERTS (still = 5)

Recommendation — Option A: Store original owner in TicketInfo

struct TicketInfo {
    address originalOwner;  // ← add this
    ...
}

function burn(uint256 tokenId) external onlyOwner {
    delete userRoundTicket[tickets[tokenId].originalOwner][tickets[tokenId].roundId];
    delete tickets[tokenId];
    _burn(tokenId);
}

Recommendation — Option B: Make tickets non-transferable

function _update(address to, uint256 tokenId, address auth)
    internal override returns (address) {
    address from = _ownerOf(tokenId);
    require(from == address(0) || to == address(0),
            "TicketNFT: non-transferable");
    return super._update(to, tokenId, auth);
}
M-2 MEDIUM ROUND_DURATION is Hardcoded as 1 Second (Unusable in Production) ▼
MoneyMoneyCome.sol : 87 Category: Deployment Risk

Description

ROUND_DURATION = 1 is a Solidity constant, which means it is baked into the bytecode at compile time and cannot be changed after deployment. Deploying to mainnet with this value makes the protocol non-functional.

uint256 public constant ROUND_DURATION = 1;  // ← 1 second, not 1 week

Recommendation

uint256 public roundDuration = 7 days;  // mutable, sensible default

function setRoundDuration(uint256 newDuration) external onlyOwner {
    require(newDuration >= 1 days, "Too short");
    roundDuration = newDuration;
}

L-1 LOW Squad Prize Distribution Integer Dust Permanently Locked ▼
SquadRegistry.sol : 152 Category: Precision Loss

Description

Integer division in the squad prize loop truncates remainders. The sum of all member payouts may be slightly less than the 20% squad pool, with the dust permanently trapped in the contract.

otherAmounts[idx] = (squadPool * memberWeights_val[i]) / totalOtherWeight;
// Up to (MAX_MEMBERS - 1) wei lost per round due to rounding down

Recommendation

// Assign remainder to the last member
uint256 distributed;
for (uint256 i; i < otherCount - 1; i++) {
    otherAmounts[i] = (squadPool * memberWeights_val[...]) / totalOtherWeight;
    distributed += otherAmounts[i];
}
otherAmounts[otherCount - 1] = squadPool - distributed;
L-2 LOW totalSupply() Counts Burned Tokens ▼
TicketNFT.sol : 117 Category: Incorrect View

Description

_nextTokenId only increments on mint and is never decremented on burn. totalSupply() returns the historical minted count, not current circulating supply.

Recommendation

uint256 private _burnedCount;

function totalSupply() external view returns (uint256) {
    return _nextTokenId - _burnedCount;
}
// In burn(): _burnedCount++;
L-3 LOW Squad Leader Field Not Updated When Leader Leaves ▼
SquadRegistry.sol : _removeMember() Category: Stale State

Description

When a squad leader calls leaveSquad(), they are removed from the members array but Squad.leader continues to point to their (now departed) address. getSquad() returns stale leader info.

Recommendation

// In _removeMember(), after removing from array:
if (s.leader == member && s.members.length > 0) {
    s.leader = s.members[0];
}
L-4 LOW _harvestYield() Has Minor Accounting Precision Loss ▼
MoneyMoneyCome.sol : 444–454 Category: ERC-4626 Rounding

Description

ERC-4626 specifies that previewWithdraw() rounds up to protect the vault. The actual USDC received on redemption may be slightly more than interest, but userKeeps and toPool are still calculated from the original estimate. The epsilon remains in the contract indefinitely.

Recommendation

uint256 actualReceived = vault.redeem(sharesToRedeem, address(this), address(this));
uint256 userKeeps = (actualReceived * retainBps) / BPS_DENOM;
uint256 toPool    = actualReceived - userKeeps;  // uses actual, not estimated
ℹ️ Informational
I-1 INFO No Emergency Pause Mechanism ▼

The contracts have no pausable functionality. If a live exploit is discovered, there is no way to halt the protocol. Recommendation: inherit OpenZeppelin's Pausable and add whenNotPaused to enterGame() and withdraw().

I-2 INFO Infinite USDC Approval to Aave Pool ▼

usdc.approve(_aavePool, type(uint256).max) is set in the YieldVault constructor. This is standard DeFi practice, but means that if the Aave pool address is ever compromised, all vault USDC could be drained. The Aave pool address must be carefully verified before deployment.

I-3 INFO performUpkeep() Has No Access Control (By Design) ▼

performUpkeep() can be called by any address, not just the Chainlink Automation registry. This is intentional for Chainlink Automation compatibility. State guards prevent exploitation in the normal case (see H-3 for the edge case where this becomes a risk).

I-4 INFO Contracts Are Not Upgradeable ▼

All contracts use standard non-proxy deployment. Any bug fix requires redeploying the full contract system and migrating user state. Consider using the UUPS or Transparent Proxy pattern (OpenZeppelin), or document a clear migration procedure for users.
